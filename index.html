Introducción

Cuando un programa termina de compilarse, su historia no acaba: el entorno en tiempo de ejecución (runtime) se encarga de cargar el binario o el bytecode, gestionar memoria, resolver llamadas dinámicas y, en muchos casos, aplicar optimizaciones adicionales mientras el programa ya está en marcha.

En paralelo, durante la compilación, la fase de optimización de código transforma la representación intermedia o el código objeto para que el programa sea más rápido, ocupe menos memoria y consuma menos recursos, sin cambiar lo que el programa calcula.

Este blog resume ambos frentes a nivel general y profundiza en las técnicas clásicas de optimización que estudian los compiladores modernos.

Entornos en tiempo de ejecución

Un entorno de ejecución es el conjunto de servicios, bibliotecas y convenciones que permiten que un programa compilado o interpretado funcione correctamente en una máquina concreta. No es solo “el sistema operativo”: incluye el runtime del lenguaje, el recolector de basura, el despachador de excepciones, la carga dinámica de módulos y, a menudo, un optimizador en caliente.

Ejecución nativa

El compilador genera código máquina (por ejemplo, con GCC o Clang). El runtime es mínimo: enlace con la libc, arranque (_start), pila y heap gestionados por el SO. La optimización pesada ocurre en compilación.

Máquinas virtuales

Java (JVM), C# (CLR) o WebAssembly usan bytecode intermedio. El runtime interpreta, compila just-in-time (JIT) o combina ambos. Aquí el optimizador puede reescribir bucles calientes después de perfilar el programa en producción.

Interpretación + runtime embebido

Python, JavaScript o Ruby delegan gran parte de la semántica al intérprete o motor (CPython, V8). El “compilador” del usuario es a menudo un transpilador; las optimizaciones críticas viven en el motor en tiempo de ejecución.

Relación con los optimizadores

Los optimizadores pueden actuar en distintos momentos:

  • En compilación — sobre código fuente, IR o código objeto (enfoque de este módulo).
  • En enlace — el linker elimina símbolos muertos y fusiona secciones (LTO).
  • En JIT — el runtime recompila métodos frecuentes con información de perfiles.

La regla de oro es la misma en todos los casos: toda transformación debe preservar la semántica observable del programa original.

Optimización de código en diseño de compiladores

La optimización de código es el proceso de mejorar un programa en velocidad, tamaño y uso de recursos manteniendo su funcionalidad. Los aspectos que el compilador suele perseguir incluyen:

  • Mayor rendimiento y menor consumo de memoria
  • Código más compacto y portable entre plataformas
  • Menor consumo energético en dispositivos embebidos
  • Tiempo de compilación razonable (no optimizar en exceso)
  • Corrección garantizada: el comportamiento lógico no cambia
  • Complejidad manejable para depuración y mantenimiento

¿Cuándo y por qué optimizar?

En la práctica, la optimización agresiva del compilador suele aplicarse al final del pipeline de desarrollo, porque puede reducir la legibilidad del IR generado y dificultar la depuración.

¿Por qué delegarla al compilador? La optimización de algoritmos (elegir un mejor algoritmo O(n log n) en lugar de O(n²)) es responsabilidad del programador. En cambio, el compilador optimiza el código intermedio: propaga constantes, elimina subexpresiones comunes y reordena instrucciones según la arquitectura destino — tareas repetitivas y propensas a error si se hicieran a mano.

Beneficios típicos: menos memoria, ejecutables más pequeños, mayor velocidad de ejecución y optimización sistemática sin reescribir el fuente línea por línea.

Tipos de optimización

Independiente de máquina

Mejora el código intermedio sin conocer registros ni direcciones absolutas de la CPU destino. Ejemplos: propagación de constantes, eliminación de código muerto, inline de funciones a nivel IR.

Dependiente de máquina

Se aplica sobre el código objeto o ensamblador ya generado: asignación de registros, selección de instrucciones, aprovechamiento de la jerarquía de caché y del pipeline del procesador.

Técnicas principales de optimización

A continuación se resumen las transformaciones más estudiadas en la fase de optimización. Cada bloque muestra el patrón “antes / después” que el optimizador busca reconocer.

1. Evaluación en tiempo de compilación

Expresiones con operandos constantes se reducen durante la compilación.

// Antes
A = 2 * (22.0 / 7.0) * r

// Después
A = 6.2857 * r

2. Propagación de variables

Sustituye una variable por otra que tiene el mismo valor en ese punto del programa.

// Antes
x = a
y = x + 5

// Después
y = a + 5

3. Propagación de constantes

Reemplaza variables por valores constantes conocidos en compile-time o por análisis de flujo.

// Antes
a = 10
b = a + 5

// Después
a = 10
b = 10 + 5

4. Plegado de constantes (constant folding)

Calcula expresiones constantes antes de generar código.

#define k 5
x = 2 * k    // → x = 10
y = k + 5    // → y = 10

5. Propagación de copias

Extensión de la propagación de constantes: si x solo copia a, usa a directamente.

// Antes
c = a * b
x = a
// ... hasta ...
d = x * b + 4

// Después
d = a * b + 4

6. Eliminación de subexpresiones comunes (CSE)

Evita recalcular la misma expresión si sus operandos no han cambiado.

// Antes
t1 = a * b
t2 = a * b + c

// Después
t1 = a * b
t2 = t1 + c

7. Eliminación de código muerto

Suprime asignaciones y cálculos cuyos resultados nunca se usan.

// Antes
x = 10
y = 20
print(y)

// Después
y = 20
print(y)

8. Eliminación de código inalcanzable

Quita instrucciones que no pueden ejecutarse (por ejemplo, después de un return).

int main() {
  int num = 10;
  cout << "GFG!";
  return 0;
  cout << num;  // inalcanzable — se elimina
}

9. Inline de funciones

Sustituye la llamada por el cuerpo de la función, eliminando overhead de llamada.

// Antes: sum(a,b) { return a + b; }
//        x = sum(2, 3);

// Después
x = 2 + 3

10. Clonación de funciones

Crea versiones especializadas según el contexto de uso (distinto del overloading del lenguaje). Útil cuando los tipos de los argumentos son conocidos en el sitio de llamada. 

// Antes: una sola add() para int y float

// Después
int add_int(int a, int b) { return a + b; }
float add_float(float a, float b) { return a + b; }

11. Variable de inducción

En bucles, mantiene una variable que evoluciona de forma predecible (por ejemplo, j = 2*i).

// Antes
for (i = 0; i < n; i++)
    j = 2 * i;

// Después
j = 0;
for (i = 0; i < n; i++)
    j = j + 2;

12. Reducción de fuerza

Reemplaza operaciones costosas por equivalentes más baratas.

// Antes:  x = y * 2
// Después: x = y + y

Optimización de bucles

Los bucles concentran gran parte del tiempo de CPU en programas numéricos y de sistemas. Tres técnicas clásicas son:

Code motion (reducción de frecuencia)

Mueve invariantes del cuerpo del bucle hacia fuera.

// Antes
for (i = 0; i < n; i++) {
    x = a * b;
    y[i] = x + i;
}

// Después
x = a * b;
for (i = 0; i < n; i++)
    y[i] = x + i;

Loop jamming (fusión de bucles)

Combina bucles con el mismo límite para reducir overhead de control.

// Antes: dos bucles sobre i
// Después: un solo bucle que actualiza a[i] y d[i]

Desenrollado de bucles (loop unrolling)

Ejecuta varias iteraciones por vuelta, reduciendo saltos y comparaciones.

// Antes
for (i = 0; i < 4; i++)
    a[i] = a[i] * 2;

// Después
a[0] = a[0] * 2;
a[1] = a[1] * 2;
a[2] = a[2] * 2;
a[3] = a[3] * 2;

¿Dónde aplicar la optimización?

Nivel Quién optimiza Ejemplos
Código fuente Programador Mejor algoritmo, estructuras de datos, refactor de bucles
Código intermedio Compilador CSE, propagación, inline, eliminación de código muerto
Código objeto Compilador (backend) Registros, selección de instrucciones, scheduling

Conclusiones

Los entornos en tiempo de ejecución y los optimizadores de compilador son piezas complementarias de un mismo objetivo: ejecutar programas de forma eficiente y correcta. El programador elige algoritmos y estructuras; el compilador aplica transformaciones mecánicas y verificables sobre el IR; el runtime puede seguir optimizando en JIT cuando dispone de información de perfiles reales.

Comprender estas técnicas — desde la propagación de constantes hasta el desenrollado de bucles — permite leer dumps de IR, interpretar flags como -O2 o -O3 en GCC/Clang y diseñar código fuente que no luche contra el optimizador.
